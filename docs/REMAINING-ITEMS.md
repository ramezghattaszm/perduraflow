Read-only for the derived part; the seeded baseline stays editable on the Operators form.

- **Graduated autonomy by measured outcome** вЂ” the confidenceГ—tier autonomy must be EARNED from measured accuracy (a tier-1 param auto-commits only after the model proves itself on that param class). Production: the mechanism that ratchets autonomy up as outcomes validate it.
- **Optimizer selection (AQ6)** вЂ” the full CP-SAT/OR-Tools optimizer (separate from ML, but the other "real engine" swap) drops in behind the `external_solver` binding; engine selection is an open architecture question.

### Actuals rollup вЂ” versionMetrics deliberately not moved (DECISION, not a gap)
> When the continuous actuals folds were centralized into `ActualsRollupService` (commit 9ea0ee1), `versionMetrics` (the per-version scorecard fold) was **intentionally left in `SchedulingService`** вЂ” this is a settled decision, not an oversight.
- **Why it's fine where it is:** it reads actuals via the `learning.read` **contract**, never raw learning tables вЂ” so O1/O2 already hold. The goal was "nothing reads raw actuals *tables* + duplicated math gets one home," not "physically relocate every actuals read." Both the cockpit path (now via the rollup) and the per-version path (via the contract) satisfy that.
- **Why moving it would be worse:** it's **single-use** (not duplicated like the cockpit folds в†’ no DRY gain) and it's **entangled with `latenessChainsFor`** (a lateness-narration helper shared by workList + 2 version-detail methods, 4 callers total). Moving it cleanly would drag a non-actuals helper into the "actuals rollup," **muddying the boundary** the refactor exists to keep clean, and touch ~8 unrelated call-sites under the parity gate for no functional gain.
- **Do not "tidy" this later** by moving it into the rollup вЂ” that reintroduces the boundary muddiness (a lateness-narration helper inside an actuals rollup) for zero functional gain. If a genuine future need ever makes the per-version fold *duplicated*, revisit then вЂ” not before. The code comment at `versionMetrics` in `SchedulingService` records the same decision on-site.

### Yield / Quality module (NOT modeled вЂ” the highest-value post-demo gap)
> The engine assumes **100% yield**. There is no Quality module. This is the single most consequential product gap identified вЂ” it is *both* a correctness gap *and* a missing predictive domain, and it is what a Tier-1 ops person would feel first in a pilot (floor reality is never 100% good).
- **Correctness gap (current):** planned **good-output quantity is overstated** вЂ” the plan schedules nominal quantity with no scrap/rework accounting, so true completion, true throughput, and downstream material consumption are all computed against an unreal 100%-good assumption. Real plans routinely miss because of this.
- **OEE Quality leg is SEEDED, not MODELED:** OEE = Availability Г— Performance Г— Quality (api-spec В§12.6). For the demo the Quality leg is honest-but-synthetic вЂ” seeded `historical_outcome` quality + `scrap_qty` injected into `execution_actual` rows so per-version Quality computes. **True OEE depends on a real yield model;** the demo's Quality number is a stand-in, the same way the wear predictor is an OLS stand-in.
- **Missing predictive domain:** yield/scrap prediction (forecast a run's defect rate *before* it runs вЂ” this part, this line, this speed, historically scraps X%) is the Quality module's predictive job (D3 prediction target #4, scrap/yield). Lets the plan account for rework loops and real good-output, not nominal. **Not built.**
- **Scope when built:** a Quality domain that (a) owns scrap/yield data + process capability, (b) feeds *expected good-output = qty Г— yield* into planning (fixing the 100% assumption), (c) computes the OEE Quality leg from real measured quality, (d) predicts yield forward. The scheduler **consumes** expected-yield through a seam (same consumed-input pattern as material/operators) вЂ” it does not own quality.
- **Likely the #1 post-demo build.**

### Scheduler hard-constraint gaps (AUDITED this cycle вЂ” evidence in `CONSTRAINT-INVENTORY.md`)
> The constraint audit (read-only, code-verified) flipped several spec'd constraints from "questionable" to confirmed-**NOT built**. Captured here as evidenced gaps, not assumptions.
- **D28 sequencing / campaign rules вЂ” the #1 hard-constraint gap (NOT built, confirmed).** None of the four legality types exist: **required-ordering, contiguity, forbidden-transition (+ required cleanout), max-consecutive.** Changeover attributes are *modeled, not sequenced* (routing schema, SKIP-48); the only attribute-aware logic is a soft, firm-excluded forecast changeover-grouping bonus (`CHANGEOVER_BONUS_HOURS=24`). **This is a CORRECTNESS gap, not just a feature gap** вЂ” in automotive a wrong transition (e.g. paint darkв†’light without a purge, or skipping a cleanout) = **scrap**. Likely ranks **with the Yield/Quality module** for post-demo priority. **Demo-honesty implication:** claim *"reduces changeovers"* (the flat switch-count IS in the objective вњ…), **never** *"enforces sequencing rules"* (it doesn't).
- **D8 per-pair changeover cost matrix (NOT built).** Changeover is in the objective only as a **flat switch-count** (weight 1) вЂ” every transition costs the same. The attribute-keyed per-pair cost matrix (transition Aв†’B costs more than Cв†’D; sequence-dependent setup time) is not built. Build alongside the D28 campaign/changeover work (same domain: the sequencer becomes attribute-aware for both legality and cost).
- **Schedule-commit approval gate (SKIP-46, NOT built).** The confidenceГ—tier autonomy gate is **LIVE for learning predictions** (`gateDisposition` вЂ” tier Г— computed confidence auto-commits/queues wear forecasts), but `commit()` on a schedule version has **no approval check** (explicit SKIP-46 seam). **Talk-track:** lean the graduated-autonomy story on the **prediction path** (genuinely live, demoable), not on schedule-commit gating (which is a future seam). Approval *tiers* are seeded (Planner/Supervisor/Plant manager) but no rule routes a commit to them yet.

### Prediction ownership by module (the platform predictive roadmap вЂ” see `PREDICTION-OWNERSHIP-BY-MODULE.md`)
> Organizing principle: **prediction belongs to the module that owns the predicted thing; the scheduler consumes forecasts through stable seams and predicts only about the PLAN.** The vision is not "the scheduler predicts more things" вЂ” it's "each domain module gains predictive intelligence; the scheduler integrates ever-richer forecasts through the seams it already uses (material, operators, downtime)." Each is the same `observe-actuals в†’ forecast в†’ confidence-gate в†’ propose в†’ adopt-on-confirm` loop the tool-wear lifecycle already implements. Full detail in the standalone doc; summarized here so it lives in the master backlog (per the all-deferred-items-captured convention).
- **Demand / Net-Requirements** в†’ demand forecasting (OEM release projection, CUM/netting вЂ” the central JIS/JIT problem). Net-requirements module is spec-only (see below).
- **Supply / Inventory** в†’ material-arrival prediction (supplier-reliability forecast of `availableAt` with a confidence band) + supply-risk flagging. Material gate built (seeded `availableAt`); prediction is future.
- **Workforce** в†’ operator-performance *derivation* from actuals + performance-trajectory forecasting (learning curve, fatigue trend). Performance is *consumed* today (C5); derivation is in the ML-engine TODO.
- **Quality** в†’ yield/scrap prediction (the Yield/Quality module above). The highest-value gap.
- **Asset / Maintenance** в†’ equipment-failure prediction в†’ predicted downtime (same shape as injected `resource_downtime`, so a line-down becomes *predicted and pre-positioned*, not just reacted to); die-life; **tool-wear**. *Boundary note:* tool-wear is currently the **scheduler's** headline predictive beat, but architecturally belongs to **Asset** (predict wear в†’ write a cycle-adjustment the scheduler consumes, like material/operators/downtime). Cleanup deferred; flagged so the boundary is intentional.
- **Scheduler (the integrator)** predicts only about the plan it owns вЂ” at-risk, what-if, cascade/ripple, bottleneck-migration, schedule-stability. Stays deterministic + authoritative; ML/forecasts feed it, never decide for it; the human confirms consequential changes.
- **Platform thesis:** PerduraFlow is a manufacturing-ops platform of cohesive domain modules, each owning its data + prediction, feeding a deterministic scheduling kernel that integrates them into one optimized, auditable, human-confirmed plan. Modular (adopt the scheduler first; add domain predictors over time). The architectural test for any new predictive feature: *which domain owns the predicted thing?* вЂ” that is where its prediction lives; anything genuinely ambiguous defaults to the scheduler.
