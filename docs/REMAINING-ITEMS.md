### Override "pin" / standing veto (autonomy-config / override family тАФ DEFERRED DECISION)
> **What it is.** A *per-lane (per resource├Чparam) standing veto*: once a human overrides a wear forecast, the confidence├Чtier gate is **disabled for that lane until the human re-approves** тАФ auto-commit can't fire again on its own, regardless of how confident the next forecast is. Distinct from today's *one-shot* override.
- **What ships today (intentional, NOT a pin).** `revertPrediction` + `dismissPrediction` are **one-shot**: they undo/snooze the current forecast and re-arm only on *materially-worse* new data (the supersede-on-worsen path). `reopenPrediction` (the manual path back) **deliberately does NOT pin** тАФ it returns the forecast to `queued` as a proposal, and a subsequent *confident* actual can re-adopt via the auto-gate **by design** (manual reconsideration vs new information are different triggers). See the re-open / revert / set-aside work (commits `72ac064`, `ce01f3c`, `b2db8e5`) and `requireAdopted`/`requireSetAside` in `learning.service.ts`; the no-pin behavior is asserted in `reopen.spec.ts` (revertedтЖТqueued, never an auto re-adopt at re-open time).
- **Pull-forward trigger.** A client asks *"why did it re-act after I said no?"* тАФ i.e. they expected an override to *stick*, not to be re-decided by the next confident reading. That's the signal the one-shot model isn't enough and a standing veto is wanted.
- **Where it belongs.** The **autonomy-config / override family** тАФ a per-scope (lane/param) flag alongside `tier1AutoThreshold` / `boundedAuto` (autonomy is already L3 tenant-only per `CONSTRAINT-INVENTORY.md`; this would be a finer-grained, lane-level override). - **Why deferred.** The one-shot override already gives the human a clean escape hatch + a manual way back; the standing veto is only warranted once the re-adopt-on-new-info behavior is actually felt as wrong in front of a client. Revisit then.

### Yield / Quality module (NOT modeled тАФ the highest-value post-demo gap)
> The engine assumes **100% yield**. There is no Quality module. This is the single most consequential product gap identified тАФ it is *both* a correctness gap *and* a missing predictive domain, and it is what a Tier-1 ops person would feel first in a pilot (floor reality is never 100% good).

### Prediction ownership by module (the platform predictive roadmap тАФ see `PREDICTION-OWNERSHIP-BY-MODULE.md`)
> Organizing principle: **prediction belongs to the module that owns the predicted thing; the scheduler consumes forecasts through stable seams and predicts only about the PLAN.** The vision is not "the scheduler predicts more things" тАФ it's "each domain module gains predictive intelligence; the scheduler integrates ever-richer forecasts through the seams it already uses (material, operators, downtime)." Each is the same `observe-actuals тЖТ forecast тЖТ confidence-gate тЖТ propose тЖТ adopt-on-confirm` loop the tool-wear lifecycle already implements. Full detail in the standalone doc; summarized here so it lives in the master backlog (per the all-deferred-items-captured convention).
- **Demand / Net-Requirements** тЖТ demand forecasting (OEM release projection, CUM/netting тАФ the central JIS/JIT problem). Net-requirements module is spec-only (see below).
- **Supply / Inventory** тЖТ material-arrival prediction (supplier-reliability forecast of `availableAt` with a confidence band) + supply-risk flagging. Material gate built (seeded `availableAt`); prediction is future.
- **Workforce** тЖТ operator-performance *derivation* from actuals + performance-trajectory forecasting (learning curve, fatigue trend). Performance is *consumed* today (C5); derivation is in the ML-engine TODO.
- **Quality** тЖТ yield/scrap prediction (the Yield/Quality module above). The highest-value gap.
- **Asset / Maintenance** тЖТ equipment-failure prediction тЖТ predicted downtime (same shape as injected `resource_downtime`, so a line-down becomes *predicted and pre-positioned*, not just reacted to); die-life; **tool-wear**. *Boundary note:* tool-wear is currently the **scheduler's** headline predictive beat, but architecturally belongs to **Asset** (predict wear тЖТ write a cycle-adjustment the scheduler consumes, like material/operators/downtime). Cleanup deferred; flagged so the boundary is intentional.
- **Scheduler (the integrator)** predicts only about the plan it owns тАФ at-risk, what-if, cascade/ripple, bottleneck-migration, schedule-stability. Stays deterministic + authoritative; ML/forecasts feed it, never decide for it; the human confirms consequential changes.
- **Platform thesis:** PerduraFlow is a manufacturing-ops platform of cohesive domain modules, each owning its data + prediction, feeding a deterministic scheduling kernel that integrates them into one optimized, auditable, human-confirmed plan. Modular (adopt the scheduler first; add domain predictors over time). The architectural test for any new predictive feature: *which domain owns the predicted thing?* тАФ that is where its prediction lives; anything genuinely ambiguous defaults to the scheduler.
